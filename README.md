# Meta-Labeling-CTTS

Meta-Labeling-CTTS packages the second-stage ("meta") classifier that sits on top of first-stage directional models (M1). It consumes the provider CSVs (`*_up.csv`, `*_down.csv`) generated by the Chronos/Bolt/Time-LLM/Tirex/TimeGPT pipelines, creates temporal windows, trains the CTTS (CNN Transformer-based classifier) (M2), and exports enriched prediction tables plus diagnostic plots.

## Data Expectations

All data lives under `CTTS/Data/<provider>/<market>/<symbol>/`. Training consumes the DOWN and UP CSVs produced by the given M1 model.

### Source CSV columns (`*_down.csv` and `*_up.csv`)

| Column | Required in | Purpose |
| --- | --- | --- |
| `date` | Both | Timestamp (parsed as UTC-aware `datetime`). Determines window order.
| Feature columns (e.g. `close`, `open`, `high`, `low`, `volume`, …) | Both | Raw price/feature series. The set used is controlled by `column_features` in the config. Missing features are filled with `NaN` after merge.
| `meta_label` | Both | Binary target produced by meta-labeling (1 = true positive or true negative). Becomes `isTP_DN` or `isTP_UP` in the merged frame.
| `pred` | Both | Discrete signal (1/0) from the first-stage model (M1). Renamed to `M1_DN`/`M1_UP`.
| `prediction` | Typically, `_down.csv` or `_up.csv` | Numerical score emitted by M1. Renamed to `M1_Prediction`.
| `pred_proba` | Optional | Probability score from M1; used when exporting diagnostics.
| `ground_truth` | Optional | Original price asset label used by M1. Propagated to output CSV when available.
| `lab` | Optional | True Label price direction column; preserved for consensus/confusion-matrix checks.
| `meta_target` | Optional | Meta-Label used for CTTS training, in files denoted as `meta_label`.

Anything beyond these is left untouched, but the meta-label training only relies on the columns above.

### Exported meta-label prediction CSVs

The meta-labeling consensus file written by [`export_predictions`](CTTS/test_utils.py#L108) (CTTS → `Output/Usual/<SYMBOL>/Run/`) contains:

| Column | Meaning |
| --- | --- |
| `date` | End-of-window date aligned with the test set.
| `prediction` | The M1 numerical prediction for the active task (`UP` or `DN`) when available; otherwise falls back to whichever side provides a value.
| `m1_pred_up` / `m1_pred_down` | Original discrete M1 signals for both tasks (if present in the source CSVs).
| `m1_pred_proba_up` / `m1_pred_proba_dn` | M1 probability scores (if present).
| `meta_label` | Meta-label ground truth for the active task (duplicated from `meta_label_up` or `meta_label_dn`).
| `ground_truth` | Upstream label matching the active task.
| `lab` | Provider-supplied label used for consensus visualisation.
| `m2_pred_up` or `m2_pred_dn` | CTTS (M2) discrete output for the active task.
| `m2_prob_up` or `m2_prob_dn` | Optional CTTS probability (only present when the loss provides logits that can be turned into probabilities).

Consensus plots (`meta_labeling_results.png`) are generated from these columns to highlight where M1 and M2 agree on a positive event.

## Training Pipeline Overview

1. **Configuration** – `train.py` loads [`config.yaml`](CTTS/config.yaml) and splits the settings into architecture (`model_up`/`model_dn`) and training (`train_up`/`train_dn`) packets.
2. **Data merge** – [`merge_meta_targets`](CTTS/data_preprocessing.py#L13) reads `${symbol}_down.csv` and `${symbol}_up.csv`, aligns them on `date`, renames core columns, and outputs a feature-rich frame with both UP/DN meta-labels.
3. **Windowing** – [`prepare_dataset`](CTTS/data_preprocessing.py#L45) converts the merged frame into sliding windows with per-window Min-Max normalisation and optional context features.
4. **Splits & weighting** – [`build_loaders`](CTTS/data_preprocessing.py#L146) creates train/val/test dataloaders, computes task-specific class weights, and returns the loss function configured for the requested target.
5. **Model instantiation** – [`CTTSModel`](CTTS/model.py#L431) couples RevIN + CNN tokeniser with Learnable Positional Encoding + Transformer encoder + classification head with pooling.
6. **Optimisation loop** – [`epoch_loop`](CTTS/train_utils.py#L19) and scheduler utilities apply AMP, gradient clipping, and compute metrics. Early stopping is handled by [`EarlyStopping`](CTTS/model.py#L7).
7. **Evaluation & exports** – [`test_utils`](CTTS/test_utils.py) routines generate confusion matrices, probabilities, and the meta-label consensus CSV/plot.

## Module Guide

### `CTTS/data_preprocessing.py`

- [`merge_meta_targets`](CTTS/data_preprocessing.py#L13): Loads UP/DN CSVs, selects requested features, renames core columns (`isTP_*`, `M1_*`), outer-joins on `date`, and optionally saves the merged view.
- [`prepare_dataset`](CTTS/data_preprocessing.py#L45): Builds sliding windows of length `seq_len`, applies per-window MinMax scaling, appends context features, and returns a `TensorDataset` of `(X, y_up, y_dn)` tensors.
- [`FocalLoss`](CTTS/data_preprocessing.py#L87): Standalone focal-loss implementation supporting binary or multi-class logits with configurable `gamma`/`alpha`.
- [`make_criteria`](CTTS/data_preprocessing.py#L124): Creates the UP/DN loss objects (BCE, CrossEntropy, or Focal) and applies the computed class weights.
- [`build_loaders`](CTTS/data_preprocessing.py#L146): Splits the dataset into train/val/test (with optional rolling cross-validation), computes class weights on the training portion, and returns `(train_loader, val_loader, criterion)` tuples plus the fixed test loader.

### `CTTS/model.py`

- [`EarlyStopping`](CTTS/model.py#L7): Monitors validation loss, supports patience, delta threshold, and an optional hard stop when the loss explodes.
- [`RevIN`](CTTS/model.py#L41): Reversible instance normalisation in the time dimension with optional affine parameters.
- [`PositionalEncoding`](CTTS/model.py#L57): Classic sinusoidal positional encoding.
- [`LearnablePosEnc`](CTTS/model.py#L84): Learnable positional embeddings initialised from the sinusoidal table with configurable dropout.
- [`CNNEncoder`](CTTS/model.py#L205): Stack of 1D convolutions (with optional SAME padding) that tokenises the sequence and injects positional encodings.
- [`TransformerEncoder`](CTTS/model.py#L273): Wrapper around `nn.TransformerEncoder` with configurable depth, heads, feed-forward size, dropout, and activation.
- [`AttentionPooling`](CTTS/model.py#L318) / [`MeanMaxPooling`](CTTS/model.py#L332): Temporal pooling strategies used by the classifier head.
- [`ClassificationHead`](CTTS/model.py#L342): Applies the chosen pooling, MLP with activation/dropout, and outputs class logits.
- [`CTTSModel`](CTTS/model.py#L431): End-to-end architecture combining RevIN → CNNEncoder → TransformerEncoder → ClassificationHead. Parameters include convolutional stack, transformer spec, classifier settings, number of classes, and window length.

### `CTTS/optim_utils.py`

- [`Lookahead`](CTTS/optim_utils.py#L10): Wrapper implementing the Lookahead optimiser on top of any base optimiser.
- [`get_optimizer`](CTTS/optim_utils.py#L45): Factory that instantiates Adagrad, RMSprop, Adam, AdamW, AdaBelief, or Ranger using the section of the config passed in.
- [`make_scheduler`](CTTS/optim_utils.py#L107): Builds warm-up, cosine, plateau, power, exponential, or composite schedulers based on the `scheduler` block.
- [`step_scheduler`](CTTS/optim_utils.py#L197): Advances the scheduler each epoch, handling warm-up dictionaries and `ReduceLROnPlateau`’s metric requirements.

### `CTTS/paths.py`

- [`dataset_path`](CTTS/paths.py#L11): Resolves `Data/<provider>/<market>/<symbol>/<symbol>_<split>.csv` and validates that the file exists.
- [`bolt_path`](CTTS/paths.py#L35), [`chronos_path`](CTTS/paths.py#L36), [`tirex_path`](CTTS/paths.py#L37): Convenience wrappers that fix the provider argument.

### `CTTS/test_utils.py`

- [`get_preds_and_targets`](CTTS/test_utils.py#L18): Runs a model over a loader and returns discrete predictions alongside targets.
- [`plot_cm_with_metrics`](CTTS/test_utils.py#L36): Saves a confusion matrix annotated with accuracy, precision, recall, F1, and Fβ scores.
- [`export_predictions`](CTTS/test_utils.py#L108): Aligns test windows with their `date`, merges M1 outputs, injects CTTS predictions/probabilities, and writes the enriched CSV used for consensus.
- [`get_preds_probs`](CTTS/test_utils.py#L214): Similar to `get_preds_and_targets` but also collects probabilities; supports both BCE and softmax losses.
- [`plot_meta_labeling_consensus`](CTTS/test_utils.py#L314): Builds a consensus confusion matrix (`meta_labeling_results.png`) to highlight scenarios where M1 and CTTS agree on true positives.

### `CTTS/train_utils.py`

- [`seed_everything`](CTTS/train_utils.py#L9): Sets seeds for Python, NumPy, and PyTorch (CPU/GPU) for reproducible runs.
- [`epoch_loop`](CTTS/train_utils.py#L19): Core train/eval loop with AMP, gradient clipping, metric computation, and Binary Cross-Entropy vs. Cross-Entropy branching.
- [`model_train`](CTTS/train_utils.py#L82): High-level training routine that logs to TensorBoard, applies early stopping, and returns the final validation loss.
- [`model_test`](CTTS/train_utils.py#L152): Evaluates a model on the test loader, prints a summary banner, and logs the results.

### `CTTS/train.py`

Orchestrates a full experiment:

1. Load the configuration and derive architecture/training slices for UP and DN tasks.
2. Resolve the data path (`dataset_path`) and push reproducible seeds.
3. Merge and window the dataset (`merge_meta_targets` → `prepare_dataset`).
4. Build train/val/test loaders and the appropriate criterion (`build_loaders`).
5. For each task (UP or DN):
   - Instantiate `CTTSModel` with the task-specific hyper-parameters.
   - Create the optimiser (`get_optimizer`), scheduler (`make_scheduler`), and early stopper.
   - Train until early stopping, tracking the best validation snapshot.
   - Export checkpoints, TensorBoard logs, confusion matrices, enriched predictions, and consensus plots via `test_utils` helpers.
6. Flush CUDA cache before moving to the next loop.

## Configuration Reference (`config.yaml`)

### `paths`

| Key | Description |
| --- | --- |
| `data_root` | Root directory containing provider sub-folders (default `Data`).
| `chronos_subdir` / `bolt_subdir` / `tirex_subdir` | Convenience names used to build paths if you add provider-specific logic.
| `output_root` | Base folder for checkpoints, TensorBoard logs, exported CSVs, and plots.

### `dataset`

| Key | Description |
| --- | --- |
| `source` | Provider folder under `data_root` (`Chronos`, `Bolt`, `Tirex`, …).
| `type` | Market bucket (`Crypto`, `Equities`, …); capitalisation is normalised.
| `symbol` | Asset ticker (e.g. `BTC`, `SPY`). Determines which CSVs are read and how outputs are named.

### `splits`

Train/validation/test proportions applied to the sliding windows produced by `prepare_dataset`. They must sum to 1.0.

### Sequence and context settings

- `sequence_length`: Number of historical timesteps (days) per model window.
- `future_offset`: Predition horizon used when building meta-labels.
- `column_features`: Ordered list of feature columns to pull from `${symbol}_down.csv`.
- `context_features`: Additional columns appended as constant context at the end of each window (e.g. macro indicators).
  Currently available: `rel_interval_width`, `forecast_slope`, `bowley_skewness`, `moors_kurtosis`, `tail_asymmetry`, `m1_pred_proba_dn`, `m1_pred_proba_up`, `M1_DN`, `M1_UP`, `M1_Prediction`, `isTP_DN`, `isTP_UP`.

### `training_mode`

| Key | Description |
| --- | --- |
| `cross_validation` | Toggle rolling CV inside the training split. When `false`, a single train/val split is used.
| `cross_val_props` | Proportion of the training+validation pool allocated to the rolling training window (only used when CV is enabled).
| `optuna_task` | Active task (`UP` or `DN`) for evaluation/export naming; also used by Optuna sweeps.
| `loss_function` | `bce`, `cross_entropy`, or `focal`.
| `focal_gamma` / `focal_alpha` | Parameters for focal loss (only used when `loss_function == 'focal'`).
| `num_classes` | Output dimension of the classifier head (1 for BCE, 2 for multi-class losses).
| `padding` | Whether the CNN tokeniser applies SAME padding to maintain the number of tokens.

### `model_up` & `model_dn`

Shared structure for the UP/DN heads:

- **CNN block**
  - `cnn_embed_dim`: Output channels per convolutional layer.
  - `cnn_kernel`: Kernel sizes for each convolutional layer.
  - `cnn_stride`: Strides for each convolutional layer.
  - `p_pos_drop`: Dropout applied to positional embeddings.
- **Transformer block**
  - `transformer.heads`: Multi-head attention heads.
  - `transformer.layers`: Number of encoder layers.
  - `transformer.ffn_dim`: Feed-forward inner dimension multiplier (times the final CNN channel size).
  - `transformer.dropout`: Dropout used inside the encoder layers.
  - `transformer.activation`: Activation function (`gelu`, `relu`, `silu`, `mish`).
- **Classifier block**
  - `classifier.mlp_hidden`: Hidden dimension of the MLP.
  - `classifier.mlp_dropout`: Dropout before the final linear layer.
  - `classifier.mlp_activation`: Activation function in the classifier MLP.
  - `classifier.mlp_pooling`: Temporal pooling strategy (`attention` or `meanmax`).

### `train_up` & `train_dn`

Training hyper-parameters for each task:

- **Optimizer**
  - `optimizer`: Name of the optimiser passed to `get_optimizer`.
  - `lr`, `weight_decay`: Learning rate and weight decay.
  - `eps`, `betas`, `lr_decay`, `lookahead_k`, `lookahead_alpha`, `weight_decouple`, `rectify`: Optional knobs whose meaning depends on the chosen optimiser (e.g. `betas` for Adam-family, `lookahead_*` for Ranger).
- **Core loop**
  - `batch_size`: Mini-batch size for all loaders.
  - `max_epochs`: Upper bound for training epochs per fold.
  - `patience`: Early-stopping patience (epochs with no validation improvement).
  - `bce_thr`: Threshold applied to sigmoid outputs when `loss_function == 'bce'`.
  - `fbeta`: β used in the Fβ score reported by `epoch_loop`.
- **Scheduler** (optional block; omit entirely to disable)
  - `sch_name`: One of `none`, `power`, `exponential`, `linear`, `cosine`, `plateau`, `linear_plateau`, `cosine_plateau`.
  - `warmup_epochs`: Linear warm-up duration for schedulers that support it.
  - `plateau_patience`, `plateau_factor`: Parameters for `ReduceLROnPlateau`.
  - `power_s`, `power_c`: Hyper-parameters for power decay.
  - `exp_s`, `exp_factor`: Parameters for exponential decay.

Use the same blocks for both tasks when you want to share hyper-parameters; override per-task values to specialise.

## Outputs

Training drops artefacts under `CTTS/Output/Usual/<SYMBOL>/Run/`:

- `<task>_best.pt`: Best-validation checkpoint for the active task.
- `Loss/train` & `Loss/validation` TensorBoard summaries (under `Output/Usual/Tensorboard/<SYMBOL>/`).
- `<TASK> — Validation.png` / `<TASK> — Test.png`: Confusion matrices with metrics.
- `<SYMBOL>_<TASK>_predictions.csv`: Enriched meta-label prediction table described above.
- `meta_labeling_results.png`: Consensus plot comparing M1 and CTTS for the active task.

## Running the Trainer

```bash
# Activate your environment, then run:
python CTTS/train.py
```

Before launching, ensure the corresponding `${symbol}_up.csv` and `${symbol}_down.csv` files are present in `CTTS/Data/<source>/<type>/<symbol>/` and adjust `config.yaml` to point to the correct provider/asset, features, and hyper-parameters.

For reproducibility across reruns, the scripts internally call `seed_everything(1493583942)`; update the seed there if you want different stochastic behaviour.

## `CTTS/optuna_optimization.py`

The Optuna script orchestrates large hyper-parameter sweeps:

- Loads the same `config.yaml`, interprets `train_up`/`train_dn` as priors, and exposes key architecture/training configurations to Optuna.
- Rebuilds datasets via `merge_meta_targets` and `prepare_dataset`, then mirrors the training loop from `train.py` inside each trial.
- Tracks validation metrics, applies early stopping, and reports scalar objectives back to Optuna for pruning/search.
- Supports CLI flags (`--trials`, `--seed`, `--device`) for custom runs. Invoke it as `python CTTS/optuna_optimization.py --trials 1000` once the data/config are prepared.

Use this script when you need systematic hyper-parameter exploration; afterwards, copy the suggested values into `config.yaml` before running the main trainer.
