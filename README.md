# Meta-Labeling Architecture: Secondary Model

Meta-Labeling-CTTS packages the second-stage ("meta") classifier that sits on top of first-stage directional models (M1). It consumes the provider CSVs (`*_up.csv`, `*_down.csv`) generated by the Chronos/Bolt/Time-LLM/Tirex/TimeGPT pipelines, creates temporal windows, trains the CTTS (CNN Transformer-based classifier) (M2), and exports enriched prediction tables plus diagnostic plots.

## Training Pipeline Overview

1. **Configuration load** – `train.py` reads [`config.yaml`](CTTS/config.yaml) and splits the settings into architecture (`model_up` / `model_dn`) and training (`train_up` / `train_dn`) packets.
2. **Data preparation** – [`merge_meta_targets`](CTTS/data_preprocessing.py#L13) ingests `${symbol}_down.csv` and `${symbol}_up.csv`, aligns them on `date`, renames core columns (`isTP_*`, `M1_*`), and returns the merged frame.
3. **Windowing** – [`prepare_dataset`](CTTS/data_preprocessing.py#L45) builds sliding windows of length `sequence_length`, applies per-window MinMax scaling, appends context tokens, and yields `(X, y_up, y_dn)` tensors.
4. **Loader & loss construction** – [`build_loaders`](CTTS/data_preprocessing.py#L146) splits the dataset into train/val/test (or rolling CV folds), computes class weights, and emits `(train_loader, val_loader, criterion)` tuples plus the fixed test loader.
5. **Training loop** – For each target task (as dictated by `training_mode.normal_task`):
   - Instantiate `CTTSModel` with the task-specific hyper-parameters.
   - Create the optimiser (`get_optimizer`), scheduler (`make_scheduler`), and early stopper.
   - Train until early stopping, tracking the best validation snapshot and logging to TensorBoard.
   - Evaluate on validation/test sets, exporting confusion matrices, enriched predictions, consensus plots, and checkpoints under `Output/Usual/<SYMBOL>/Run/`.
   - Flush CUDA cache before moving to the next task.
6. **Selective classification** – The final validation sweep converts probabilities into risk/coverage curves via `selective_classification.py`, applies the configured policy (`risk_budget` or `f_beta`) to pick an operating threshold, and records the associated metrics/curves for later Optuna-driven tuning.

## Data Expectations

All provider CSVs must live under `CTTS/Data/<provider>/<market>/<symbol>/<granularity>_<meta_label_mode>/`. For each granularity (e.g. `1h`, `1d`) and meta-label mode (`og`, `tp`, `fp`), the folder must contain both `${symbol}_down.csv` and `${symbol}_up.csv` generated by the upstream M1 model.

### Source CSV columns (`*_down.csv` and `*_up.csv`)

| Column | Required in | Purpose |
| --- | --- | --- |
| `date` | Both | Timestamp (parsed as UTC-aware `datetime`). Determines window order.
| Feature columns (e.g. `close`, `open`, `high`, `low`, `volume`, …) | Both | Raw price/feature series. The sets used per task are controlled by `column_features_up` / `column_features_dn` (or the shared `column_features`) in the config. Missing features are filled with `NaN` after merge.
| `meta_label` | Both | Meta-label target whose semantics depend on `training_mode.meta_label_*`. `original`/`tp` map to `isTP_*`, `fp` maps to `isFP_*`. `False`/`NaN` rows mark "not applicable" trades and are filtered after windowing.
| `pred` | Both | Discrete signal (1/0) from the first-stage model (M1). Renamed to `M1_DN` / `M1_UP`.
| `prediction` | Typically one side | Numerical score emitted by M1. Renamed to `M1_Prediction`.
| `pred_proba` | Optional | Probability score from M1; used when exporting diagnostics.
| `ground_truth` | Optional | Original price label used by M1. Propagated when available.
| `lab` | Optional | Provider-supplied direction label kept for consensus plots.
| `meta_target` | Optional | Alternate name for the meta-label column.

### Optional Chronos-derived statistics

Chronos exports quantile fields (`q10`, `q50`, `q90`, `oct1`–`oct7`) that `merge_meta_targets` can transform into higher-order descriptive features. Add any of the following names to `column_features_up` / `column_features_dn` in `config.yaml` and the loader will compute them automatically:

- `rel_interval_width`
- `forecast_slope`
- `bowley_skewness`
- `moors_kurtosis`
- `tail_asymmetry`

These augmentations are only produced when the upstream CSV contains the required quantiles—currently true for Chronos runs. Bolt, Tirex, TimeGPT, and Kronos outputs do not include those columns yet, so enabling the feature names has no effect for those providers.

### Meta-Label Modes & Cases

`meta_label` rows produced by the M1 pipelines can be interpreted in three ways. Pick the desired behaviour via `training_mode.meta_label_usual` (and its Optuna/Ablation counterparts); the value travels all the way to dataset paths, column names, and NaN filtering:

- **`original`** – legacy Chronos/Bolt behaviour (“current” TP-only gating). Columns are renamed to `isTP_UP` / `isTP_DN`.
- **`tp`** – strict true-positive mode. Only trades where M1 agreed with the realised direction keep a label of `1`; other tradable cases are `0`; non-tradable rows remain `NaN`.
- **`fp`** – false-positive blocker. Trades that M1 would have lost money on become `1` so the M2 model can veto them, while profitable trades are `0` and non-tradable rows are `NaN`. Columns are renamed to `isFP_UP` / `isFP_DN`.

## Module Guide

> **Note:** Ablation-specific and Optuna-specific modules are summarised later in dedicated sections.

### `CTTS/data_preprocessing.py`
- [`merge_meta_targets`](CTTS/data_preprocessing.py#L13): Loads UP/DN CSVs, selects requested features, renames core columns (`isTP_*` or `isFP_*` depending on the active meta-label mode), converts boolean `False` to `NaN` so “not applicable” trades survive the merge, outer-joins on `date`, and optionally saves the merged view.
- [`prepare_dataset`](CTTS/data_preprocessing.py#L45): Builds sliding windows, applies per-window scaling, appends context features, and returns a `TensorDataset`. It preserves NaNs while windowing and drops only those windows whose terminal label is missing for the requested task.
- [`FocalLoss`](CTTS/data_preprocessing.py#L87): Standalone focal-loss implementation supporting binary or multi-class logits with configurable `gamma`/`alpha`.
- [`make_criteria`](CTTS/data_preprocessing.py#L124): Creates the UP/DN loss objects (BCE, CrossEntropy, or Focal) and applies the computed class weights.
- [`build_loaders`](CTTS/data_preprocessing.py#L146): Splits the dataset, computes class weights, and returns `(train_loader, val_loader, criterion)` tuples plus the fixed test loader.

### `CTTS/model.py`
- [`EarlyStopping`](CTTS/model.py#L7): Monitors validation loss, supports patience/delta thresholds, and aborts on divergence.
- [`RevIN`](CTTS/model.py#L41): Reversible instance normalisation.
- [`PositionalEncoding`](CTTS/model.py#L57) / [`LearnablePosEnc`](CTTS/model.py#L84): Sinusoidal and learnable positional encodings.
- [`CNNEncoder`](CTTS/model.py#L205): Tokenises sequences using stacked 1D convolutions.
- [`TransformerEncoder`](CTTS/model.py#L273): Wrapper around `nn.TransformerEncoder` with configurable depth, heads, dropout, and activation.
- [`AttentionPooling`](CTTS/model.py#L318) / [`MeanMaxPooling`](CTTS/model.py#L332): Temporal pooling strategies for the classifier head.
- [`ClassificationHead`](CTTS/model.py#L342): Applies pooling + MLP to produce logits.
- [`CTTSModel`](CTTS/model.py#L431): Full architecture wiring RevIN → CNN → Transformer → Classifier.

### `CTTS/paths.py`
- [`dataset_path`](CTTS/paths.py#L11): Resolves `Data/<provider>/<market>/<symbol>/<granularity>_<meta_label_mode>/<symbol>_<split>.csv` and validates existence, so Chronos/Bolt runs with different meta-label modes never overwrite each other.
- [`bolt_path`](CTTS/paths.py#L35) / [`chronos_path`](CTTS/paths.py#L36) / [`tirex_path`](CTTS/paths.py#L37): Provider-specific wrappers.


### `CTTS/train.py`
- High-level script that executes the pipeline described above. It threads the selected `meta_label_usual` value into every helper so the correct source folder (`1h_tp`, `1h_fp`, `1h_og`, …) is read and the appropriate column aliases are used.
- Persists checkpoints, TensorBoard traces, confusion matrices, enriched predictions, and consensus plots for the task set by `training_mode.normal_task`, filtering out any windows whose terminal label is `NaN` so the model only trains on actionable trades.

### `CTTS/Utils/test_utils.py`
- [`plot_cm_with_metrics`](CTTS/Utils/test_utils.py#L18): Saves confusion matrices annotated with accuracy, precision, recall, F1, and Fβ scores.
- [`export_predictions`](CTTS/Utils/test_utils.py#L90): Aligns test windows with dates, merges M1 outputs, injects CTTS predictions/probabilities, and writes enriched CSVs.
- [`plot_meta_labeling_consensus`](CTTS/Utils/test_utils.py#L280): Produces consensus confusion matrices (`M1+M2_<TASK>_Results.png`).

### `CTTS/Utils/train_utils.py`
- [`seed_everything`](CTTS/Utils/train_utils.py#L9): Seeds Python, NumPy, and PyTorch for reproducibility.
- [`task_features`](CTTS/Utils/train_utils.py#L33): Resolves per-task feature lists with fallbacks to shared keys.
- [`epoch_loop`](CTTS/Utils/train_utils.py#L67): Unified train/val/test pass with optional AMP, gradient clipping, metric aggregation, and raw output capture.
- [`model_train`](CTTS/Utils/train_utils.py#L238): Orchestrates epoch loops, logging, and early stopping.
- [`model_test`](CTTS/Utils/train_utils.py#L319): Evaluates checkpoints on the test loader and prints a summary banner.

### `CTTS/Utils/optim_utils.py`
- [`Lookahead`](CTTS/Utils/optim_utils.py#L10): Lookahead wrapper for arbitrary optimisers.
- [`get_optimizer`](CTTS/Utils/optim_utils.py#L45): Instantiates Adagrad, RMSprop, Adam, AdamW, AdaBelief, or Ranger.
- [`make_scheduler`](CTTS/Utils/optim_utils.py#L107): Builds warm-up, cosine, plateau, power, exponential, or composite schedulers.
- [`step_scheduler`](CTTS/Utils/optim_utils.py#L197): Advances schedulers each epoch, handling warm-up and `ReduceLROnPlateau` cases.

### `CTTS/selective_classification.py`
- [`collect_risk_coverage_curve`](CTTS/selective_classification.py#L7): Sweeps thresholds to tabulate coverage, risk, and selection counts (optionally error tallies).
- [`coverage_at_risk`](CTTS/selective_classification.py#L67): Finds the highest-coverage threshold that satisfies the configured risk budget and coverage minima.
- [`area_under_risk_coverage`](CTTS/selective_classification.py#L147): Computes the AURC summary used as a Pareto objective for Optuna searches.
- [`plot_coverage_risk_curve`](CTTS/selective_classification.py#L190): Renders/exports coverage–risk curves for TensorBoard or offline inspection.
- [`save_metrics`](CTTS/selective_classification.py#L268): Serialises all threshold diagnostics to JSON for downstream analysis.

## Selective Classification

- **`train.py` integration** – After each fold, the trainer converts validation probabilities into gating scores, builds the risk–coverage curve, and applies the configured policy via `coverage_at_risk` or `select_threshold_fbeta` (`CTTS/train.py#L360`). The chosen threshold drives the two validation/test confusion matrices, the exported coverage–risk plot, and the JSON summary that captures coverage, risk, and selection counts.
- **`optuna_optimization.py` integration** – Every Optuna trial performs the same sweep using `collect_risk_coverage_curve`, logs the AURC, and stores the policy-specific `best_metric` (`CTTS/optuna_optimization.py#L360`). The `training_mode.optuna_objectives` list lets you treat those metrics as optimisation targets; by default the study minimises `mean_val_loss` while maximising the threshold-aware `best_metric`.
- **Policy semantics** – When `threshold.policy` is `risk_budget`, `best_metric` corresponds to coverage and the `floor_policy` setting enforces a minimum coverage (e.g. `0.05` forces ≥5 % of samples to be retained). Switching to `f_beta` turns `best_metric` into the Fβ score, so a `floor_policy` of `0.35` demands Fβ ≥35 %. Adjust `alpha`, `fbeta`, `gating`, `min_coverage`, and `min_selected_count` alongside the floor to express the level of conservatism you want from the gate.

## Outputs

Training artefacts are written to `CTTS/Output/Usual/<PROVIDER>/<SYMBOL>/<TASK>/<GRANULARITY>/Run__<TIMESTAMP>/` (the `<TIMESTAMP>` suffix matches the sibling TensorBoard directory):

- `<TASK>_best.pt`: Best-validation checkpoint for the active task.
- `M2_<TASK> — Validation.png` / `M2_<TASK> — Test.png`: Confusion matrices with metrics.
- `M1+M2_<TASK>_Results.png`: Consensus confusion matrix combining M1 and CTTS.
- `M2_<TASK>_Val_RiskCoverage.png`: Coverage–risk curve produced during selective classification thresholding.
- `M2_<TASK>_R&C_Analysis.json`: JSON summary capturing the selected threshold, coverage, risk, and validation/test diagnostics.
- `<SYMBOL>_<TASK>_predictions.csv`: Enriched meta-label prediction table (details below).
- TensorBoard logs stored under `CTTS/Output/Usual/Tensorboard/<PROVIDER>/<SYMBOL>/<TASK>_<TIMESTAMP>/<GRANULARITY>/Run/`.

### Exported meta-label prediction CSVs

[`export_predictions`](CTTS/Utils/test_utils.py#L108) produces a CSV containing:

| Column | Meaning |
| --- | --- |
| `date` | End-of-window date aligned with the test set.
| `prediction` | M1 numerical prediction for the active task (falls back to whichever side provides a value).
| `m1_pred_up` / `m1_pred_down` | Original discrete M1 signals for both tasks (if present).
| `m1_pred_proba_up` / `m1_pred_proba_dn` | M1 probability scores (if present).
| `meta_label` | Meta-label ground truth for the active task (`meta_label_up`/`meta_label_dn`).
| `ground_truth` | Upstream price-direction label matching the active task.
| `lab` | Provider label used in consensus/confusion plots.
| `m2_pred_<task>` | CTTS discrete output for the active task.
| `m2_prob_<task>` | CTTS probability (present when recoverable from logits).

These files feed the consensus plot and facilitate downstream analysis.

## Configuration Reference (`config.yaml`)

### `paths`
| Key | Description |
| --- | --- |
| `data_root` | Root directory containing provider sub-folders (default `Data`).
| `chronos_subdir` / `bolt_subdir` / `tirex_subdir` | Convenience names used to build provider-specific paths.
| `output_root` | Base folder for checkpoints, TensorBoard logs, exported CSVs, and plots.

### `dataset`
| Key | Description |
| --- | --- |
| `source` | Provider folder under `data_root` (`Chronos`, `Bolt`, `Tirex`, …).
| `type` | Market bucket (`Crypto`, `Equities`, …); capitalisation is normalised.
| `symbol` | Asset ticker (e.g. `BTC`, `SPY`).

### `splits`

Train/validation/test proportions applied to the sliding windows produced by `prepare_dataset`. Values must sum to 1.0.

### Sequence and context settings

- `sequence_length`: Number of historical timesteps per model window.
- `future_offset`: Prediction horizon used when building meta-labels.
- `column_features_up` / `column_features_dn`: Feature lists consumed by the UP/DN datasets.
- `context_features_up` / `context_features_dn`: Context tokens appended to each window for the UP/DN tasks.

### `training_mode`

- **Optuna block**
  - `cv_optuna`, `cross_val_props`: Control rolling CV inside the training split when running hyper-parameter sweeps.
  - `optuna_task`: Default task (`UP`/`DN`) optimised during Optuna runs.
  - `optuna_both`: When `true`, Optuna runs both tasks sequentially; otherwise it only optimises `optuna_task`.
  - `optuna_objectives`: Ordered list of `[direction, metric]` pairs parsed by `optuna_optimization.py` to build single- or multi-objective studies (defaults to minimising `mean_val_loss` and maximising `best_metric`).
  - `granularity_optuna`: Data granularity used by the Optuna sweeps (independent from the usual training granularity).
  - `meta_label_optuna`: Chooses the interpretation of the upstream meta labels (`original`, `tp`, `fp`). Drives the folder suffix (`1h_tp`, `1h_fp`, …) and the NaN filtering used by the Optuna datasets.

- **Usual training block**
  - `normal_task`: Task that `train.py` trains/evaluates when producing checkpoints and exports (falls back to `optuna_task`).
  - `cv_usual`: Enables validation folds during standard training runs (mirrors the Optuna setting when you want comparable behaviour).
  - `granularity_usual`: Primary data granularity consumed by `train.py`.
  - `meta_label_usual`: Same choices as above; determines which CSV suffix is read, whether `isTP_*` or `isFP_*` columns are created, and which windows are filtered before training.
  - `loss_function`: `bce`, `cross_entropy`, or `focal`.
  - `focal_gamma` / `focal_alpha`: Parameters for focal loss (only relevant when `loss_function == 'focal'`).
  - `num_classes`: Output dimension of the classifier head (1 for BCE, 2 for multi-class losses).
  - `padding`: Whether the CNN tokeniser applies SAME padding to maintain token counts.

- **Ablation block**
  - `ablation_task`: Task targeted by `train_ablation.py` (falls back to `normal_task` if omitted).
  - `meta_label_ablation`: Meta-label mode used while ablation runs rewrite context features; internally the script copies it into `meta_label_usual` before calling `run_training`.

- **Selective classification block**
  - `threshold.policy`: Chooses the selection strategy—`risk_budget` enforces an error-rate ceiling, whereas `f_beta` maximises the Fβ score.
  - `threshold.alpha` / `threshold.fbeta`: Policy-specific targets (`alpha` sets the maximum tolerated risk, `fbeta` sets the Fβ weighting).
  - `threshold.gating`: Decides how probabilities are translated into gating scores (e.g. `prob_only`, `positive_only_hard`).
  - `threshold.min_coverage`, `threshold.min_selected_count`: Guard rails that prevent degenerate low-volume selections.
  - `threshold.floor_policy`: Minimum acceptable value for the metric associated with `policy`—for `risk_budget` the floor applies to coverage (e.g. `0.05` means ≥5 % coverage), while for `f_beta` it bounds the Fβ score (e.g. `0.35` requires Fβ ≥35 %).



Example fragment:

```yaml
training_mode:
  optuna_objectives:
    - [minimize, mean_val_loss]
    - [maximize, best_metric]
  threshold:
    policy: risk_budget
    alpha: 0.20
    fbeta: 0.90
    gating: prob_only
    min_coverage: 0.0
    min_selected_count: 1
    floor_policy: 0.05
```

Switching `policy` to `f_beta` makes `best_metric` represent the Fβ score instead of coverage, so raise `floor_policy` accordingly (for instance `0.35` forces an Fβ ≥35 %).

### `model_up` & `model_dn`

(Identical structure for both tasks)

- **CNN block** – `cnn_embed_dim`, `cnn_kernel`, `cnn_stride`, `p_pos_drop`.
- **Transformer block** – `transformer.heads`, `transformer.layers`, `transformer.ffn_dim`, `transformer.dropout`, `transformer.activation`.
- **Classifier block** – `classifier.mlp_hidden`, `classifier.mlp_dropout`, `classifier.mlp_activation`, `classifier.mlp_pooling`.

### `train_up` & `train_dn`

- **Optimizer** – `optimizer`, `lr`, `weight_decay`, and optional optimiser-specific knobs (`eps`, `betas`, `lr_decay`, `lookahead_*`, etc.).
- **Core loop** – `batch_size`, `max_epochs`, `patience`, `bce_thr`, `fbeta`.
- **Scheduler** – Optional `scheduler` block supporting `none`, `power`, `exponential`, `linear`, `cosine`, `plateau`, `linear_plateau`, and `cosine_plateau`, plus warm-up and plateau parameters.

## Environment Setup

The project assumes a Python 3.11 environment. A quick way to reproduce the authors' setup using Miniconda is:

```bash
# 1. Install Miniconda (Linux example)
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
bash Miniconda3-latest-Linux-x86_64.sh

# 2. Initialise conda and restart your shell
~/miniconda3/bin/conda init
source ~/.bashrc  # or restart the terminal

# 3. Create and activate the environment
conda create -n CTTS python=3.11
conda activate CTTS

# 4. Install project requirements
pip install -r requirements.txt
```

The `requirements.txt` file lists the exact Python packages (PyTorch, Optuna, TensorBoard, etc.) expected by the training, ablation, and Optuna scripts.

## Running the Trainer

```bash
# Activate your environment, then run:
python CTTS/train.py
```

- `training_mode.normal_task` selects which task (`UP`/`DN`) is trained.
- `train_<task>` and `model_<task>` provide the hyper-parameters consumed by `train.py`.
- Feature selections come from `column_features_<task>` / `context_features_<task>`.

Before launching, ensure the corresponding `${symbol}_up.csv` and `${symbol}_down.csv` files are present under `CTTS/Data/...` and that `paths.output_root` points to a writable location.

For reproducibility across reruns, the scripts internally call `seed_everything(1493583942)`; update the seed if you need different stochastic behaviour.

## Ablation Study

```bash
# Activate your environment, then run:
python CTTS/train_ablation.py
```

- `training_mode.ablation_task` selects the target task for the ablation sweep (falls back to `normal_task`).
- Baseline context features per task live in `BASELINE_FEATURES_BY_TASK` inside `train_ablation.py`; adjust them if your project mandates a different core set.
- The script rewrites the relevant `context_features_<task>` entry for each combination, invokes `run_training`, and deposits results under `CTTS/Output/Ablation/<Group>/<Slug>/` along with a `metadata.json` built via `ablation_utils.write_metadata`.

## Optuna Study

```bash
# Activate your environment, then run:
python CTTS/optuna_optimization.py --trials 500
```

- `training_mode.optuna_task` decides which task is optimised when `training_mode.optuna_both` is `false`; setting `optuna_both: true` runs both tasks sequentially.
- The sweep honours `train_<task>` / `model_<task>` ranges encoded in the config and produces Pareto-optimal configs under `Output/Optuna/Task_<TASK>/run_<timestamp>/`.
- Use the exported configs to update `config.yaml` before running `train.py`.
